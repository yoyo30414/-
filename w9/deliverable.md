## 🟥 前言：AI 時代的生存法則
資工系的競爭力已不再是「我能手寫多複雜的演算法」，而是「我能如何指揮 AI 交付高品質系統」。這份建議書將協助你將課堂作業轉化為具備「AI 調度能力」的實戰作品集。

---

## 🟧 第一階段：工作流審計 —— 停止低價值的肉體勞動
*(由 Specialist 1: Workflow Auditor 診斷)*

### 1. 典型「代碼工 (Code Monkey)」行為警告
在你的作品集中，若出現以下行為，將被視為「極易被淘汰」的信號：
- **手動 CRUD 循環**：為每個資料表手動編寫重複的 DAO/Service 邏輯。
- **像素級 UI 刻畫**：手動在 CSS/XML 中調整間距與顏色，而非使用設計系統或 Agent 產出。
- **試錯式修 Bug**：複製貼上 Stack Overflow 代碼後進行無意義的微調。

### 2. AI 監督者核心量化指標
你應該開始追蹤自己的開發數據，並寫在履歷的「效能優化」部分：
- **手動代碼率 (MCR)**：< 20% (目標是讓 AI 完成 80% 的實作)。
- **審查/實作比 (RPR)**：> 4:1 (你應花更多時間在 Code Review 與架構設計)。

---

## 🟨 第二階段：戰術部署 —— 建立你的 Agent 協同防線
*(由 Specialist 2: Orchestra Commander 提供)*

### 1. 課堂作業的「雙代理人」架構
不要單純用 AI 幫你寫 code。在 GitHub 展現你的 **Agentic Workflow**：
- **執行代理人 (Implementation Agent)**：負責根據作業 Spec 產出初始代碼。
- **評審代理人 (Review Agent)**：負責根據邊界條件進行單元測試與邏輯攻擊。
- **成果**：在 README 展示「雙 Agent 循環修正日誌」，證明你具備**系統化驗證**與**自動化流程設計**的能力。

### 2. 履歷關鍵字升級
- **淘汰**：使用 Python 實現二元搜尋樹。
- **晉級**：**策劃 (Orchestrated)** 自動化開發鏈，利用多 Agent 協作將單元測試覆蓋率提升至 95%，減少 70% 的手動調試工時。

---

## 🟩 第三階段：資產固化 —— 將作業轉化為結構化知識庫
*(由 Specialist 3: Prompt Diagnostician 提供)*

### 1. 建立 RAG-Friendly 的專案結構
每個作品集專案必須包含一個 `CONTEXT.md`，供未來 AI 讀取：
- **Decision Log**：記錄為什麼選擇 A 演算法？(展示 AI 無法取代的人類決策)。
- **Reflection Snippets**：記錄 AI 的幻覺點與你的修正過程 (展示你對 Prompt 的調校能力)。

### 2. 高精準 Prompt 框架 (RCTCO)
使用以下框架來生成你的履歷與文件，避免「AI 味」：
- **Role**: 資深技術招募官。
- **Context**: 提供你的 `CONTEXT.md` 與 `Decision Log`。
- **Constraint**: 嚴禁虛詞，必須包含具體技術棧與量化指標。

3. **Day 21-30**：使用 RCTCO 框架重構履歷，將「開發者」身份升級為「AI 協作架構師」。
